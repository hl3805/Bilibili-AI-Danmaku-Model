# 第4步：数据深度清理与探索性分析 (Phase 2 - Clean & EDA)

## 一、 任务概述
本阶段你需要接管原始采集数据，执行严格的清洗、去重与多源数据匹配，建立“干净、可靠”的分析底座。随后，基于清洗后的真实数据（**严禁使用和参考前期日志中的预估数据**），进行多维度的探索性分析，重点挖掘标签体系的内部关联与头部效应。

## 二、 核心执行步骤

### 2.1 数据清理与双向匹配 (Data Cleaning & Matching)
1. **读取视频基础表**：加载原始采集的视频 CSV 数据。
2. **严格去重**：以 `bvid`（视频唯一标识）为基准，剔除所有重复记录。
3. **阈值硬过滤**：
   * 剔除 `stat_view`（播放量）严格小于 50,000 的记录。
   * 剔除 `stat_danmaku`（弹幕量）严格小于 50 的记录。
4. **弹幕文件扫描**：扫描 `danmaku/` 文件夹下所有的弹幕 CSV 文件，提取文件名对应的真实 BVID 列表。
5. **双向交集匹配**：
   * 仅保留“在视频表中存在，且同时拥有对应弹幕文件”的视频记录。
   * 仅保留“属于上述合法视频记录对应”的弹幕文件。
   * 剔除所有单边孤立的数据（有视频无弹幕、或有弹幕无视频）。

### 2.2 数据安全落盘 (Data Persistence)
1. 创建全新的目标文件夹 `cleaned-data/` 以及子文件夹 `cleaned-data/danmaku/`。
2. 将经过双向匹配清洗后的最终视频数据表，保存至 `cleaned-data/bilibili_video_cleaned.csv`。
3. 将匹配成功的弹幕文件，**复制**（Copy）到 `cleaned-data/danmaku/` 目录下。
4. **绝对红线**：全过程中，**禁止对 `data/raw/` 及原 `danmaku/` 文件夹下的任何文件进行修改、覆盖或删除操作**。

### 2.3 探索性数据分析 (EDA)
必须基于 `cleaned-data/` 中的最新文件进行实时计算：
1. **基础概览**：输出清洗后数据的 `head(5)` 样本，视频+弹幕字段代表的意思，并统计最终的“有效视频总量”。
2. ：播放量阶梯分布分析：将视频的 `stat_view`（播放量）进行分箱（Binning）统计，洞察流量的层级分布。
 **分箱规则**：
   * 级别 1：`< 50,000`（< 5万）
   * 级别 2：`50,000 - 200,000`（5万 - 20万）
   * 级别 3：`200,000 - 1,000,000`（20万 - 100万）
   * 级别 4：`> 1,000,000`（大于 100万）
**统计执行**：计算每个播放量区间内的视频数量及占比。
**可视化**：绘制播放量分布的环形图或柱状图，保存至 `results/figures/cleaned-view_distribution_bins.png`。
3. **标签大盘统计**：
   * 统计【无标签视频数量】（标签字段为空或全为空格）。
   * 提取所有视频的标签并展平统计，输出【Top 30 标签明细表】（按出现频次降序）。
   * 统计【Others 视频数量】：即该视频包含的所有标签中，没有任何一个属于 Top 30 标签集合的视频总数。
4. **标签共现与头部视频洞察（针对 Top 30 标签，逐一遍历计算）**：
   * **共现分析**：对于含有该标签的所有视频集合，统计与之共现的其他标签比例。按以下层级输出标签名及其精准百分比：
     * `> 50%` 的伴生标签列表
     * `20% - 49%` 的伴生标签列表
     * `10% - 19%` 的伴生标签列表
   * **头部效应挖掘**：在含有该标签的视频集合中，分别检索：
     * Top 3 播放量视频：输出 `BVID`、`播放量`、`弹幕量`、`发布时间`。
     * Top 3 弹幕量视频：输出 `BVID`、`播放量`、`弹幕量`、`发布时间`。
5. 对top30标签+【others视频数量】可视化，保存图片

### 2.4 生成最终报告
将上述所有 EDA 计算结果汇总，排版成结构清晰的 Markdown 文档，保存为 `cleaned-data-eda.md`（置于项目根目录或报告目录下）。

## 三、 ⚠️ Agent 注意事项与常见坑点 (Pitfalls)

1. **盲信数据的陷阱**：
   * **坑点**：前期由于采集并发或API中断，进度日志中的数字（如“总计6393条”）往往存在大量重复或无效占位。
   * **对策**：忘掉过去的日志。你的 EDA 报告中所有的 `Count`、`Sum`、`Percentage` 必须 100% 来源于你刚刚写入 `cleaned-data` 的内存 DataFrame。
2. **匹配逻辑的陷阱**：
   * **坑点**：弹幕文件名可能带有后缀（如 `.csv`），在与视频表的 `bvid` 列进行集合（Set）运算交集时，如果不去掉后缀，会导致交集为空，把有效数据全部误删。
   * **对策**：匹配前，务必对弹幕文件名进行字符串剥离操作，确保用于比对的 ID 格式完全一致。
3. **标签解析的陷阱**：
   * **坑点**：`tags` 字段通常是字符串形式（可能是逗号分隔，也可能是竖线 `|` 分隔）。如果在计算百分比时没有彻底清洗空白字符和统一分隔符，会导致同一个标签被拆分成多个（例如 `"AI"` 和 `" AI "`）。
   * **对策**：做 Explode 操作前，必须进行 `strip()` 清理，并统一采用正确的正则表达式拆分。
4. **Others 计算的逻辑陷阱**：
   * **坑点**：“Others视频数量”不是指“Top 30以外的标签出现次数”，而是指**视频实体**的去重数量。
   * **对策**：逻辑应该是判定某一行视频记录的标签列表中 `intersection(当前视频标签集合, Top30标签集合) == 空集`，满足此条件的视频行数累加。
5. **共现率分母陷阱**：
   * **坑点**：计算“带科技标签的视频，有XX%带AI标签”时，分母必须是“带有科技标签的视频总数”，而不是“全局视频总数”。
   * **对策**：必须在 groupby 或 mask 筛选出特定子集后，再进行子集内部的频次统计和除法运算。
6. **文件 I/O 与内存管理**：
   * **坑点**：几千个弹幕文件在执行 Copy 时，如果使用单线程读写可能会卡死。
   * **对策**：推荐使用 `shutil.copy` 或 `shutil.copy2` 进行底层文件级复制，切勿用 `pandas` 先 read 再 to_csv（会改变原始数据格式精度且极慢）。
