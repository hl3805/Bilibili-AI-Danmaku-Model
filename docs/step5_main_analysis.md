# 第5步：数据深度研究、假设验证与业务洞察 (Phase 3 - Deep Analysis)

## 一、 任务概述
经过前期清洗，我们已获得 6,155 条高质量的 B站 AI 相关视频及对应的弹幕数据。本阶段，Agent 需要从探索性分析（EDA）走向**因果推断、时序研究与文本挖掘**，最终产出面向业务落地的分析报告。

可以阅读results\cleaned-data-eda.md了解数据构成

将围绕 8 个独立的深度业务方向展开分析。分析的最终目的不仅是得出统计结果，更要将其转化为能指导 UP 主创作、洞察 B 站推流逻辑或反映社会情绪的“业务语言”。Agent 需要创建多个subagent分别读取子任务 Markdown 文件并执行，输出report（最多两个，主agent检查进度，在subagent结束任务后再创建新的subagent完成后续子任务）

注意：子任务md可能存在错误/不周全，在认为有必要时可以偏离子任务md具体方法，并需要在其report中注明原因。

其report需要包含使用数据，数据处理，数据分析，验证，结论，业务语言，未来方向，注意严谨性，专业性，可视化。

subagent需要每30min输出，向用户展示当前进度和已完成项；主agent需检查subagent进度，并决定是否干预（只在必要时）


## 二、 全局数据纪律与采样偏差警示 (Crucial Pitfalls)
在执行任何子任务前，必须深刻理解并处理我们数据集特有的**采样偏差（Sampling Bias）**，否则得出的结论将毫无严谨性：
1.  **配额不均陷阱**：
    * 正常标签每季度采集 10 页。
    * `科技` 标签每季度仅采集 5 页。
    * `DeepSeek` 标签仅采集了 2024 年的 4 个季度。
    * *对策*：**绝对禁止**直接比较不同标签的“绝对视频数量”。跨标签对比时，必须使用**均值、比例或标准化（Normalize）**后的指标。针对时序分析，必须按季度和采集配额进行加权调整。所有对比必须使用比例（Ratio）、均值（Mean）、比值（Rate）或标准化后的相对变化率。
2.  **模糊匹配的噪音**：
    * 数据采集中使用了模糊匹配，部分视频可能只是标题擦边（例如“科技”大类下的非AI内容，或非AI语境下的“焦虑”）。
    * *对策*：所有深入分析前，必须通过 `title` 和 `tags` 的文本相似度进行“二次提纯”，或在回归模型中加入“相关度权重”。
3.  **数据只读原则**：
    * 所有分析必须基于 `data/cleaned-data/` 中的文件。
    * **绝对禁止**修改或覆盖该目录下的任何原始清洗数据。
    * 执行中生成的中间表、特征工程数据必须存入全新的 `data/processed/step5/` 目录。
4. **容忍“无结论”**：真实世界的数据往往是嘈杂的。如果假设验证失败（$p > 0.05$）、结果无显著差异或数据不足以支撑结论，请**如实报告**。在业务中，“某种策略无效”本身就是极具价值的避坑指南。


## 三、 大规模弹幕处理工程规范 (1.5GB 数据提速与断点策略)
面对 1.5GB、数千个独立的弹幕 CSV 文件，单线程读取与清洗将耗时过长且极易崩溃。subAgent 在执行涉及弹幕分析的子任务时，**必须**遵循以下工程规范：
1. **多进程加速 (Multiprocessing)**：使用 `multiprocessing.Pool` 或 `concurrent.futures.ProcessPoolExecutor` 并行读取和预处理多个 CSV 文件，将 CPU 核心利用率打满。
2. **数据降维与分块 (Chunking)**：只读取分析所需的特定列（如仅读取 `content` 和 `ctime`）。对全量弹幕进行分词等耗时 NLP 操作时，将数据切分为 Chunk 处理，避免一次性加载爆内存。
3. **状态持久化与断点续传 (Checkpointing)**：
   * 创建本地 SQLite 数据库或使用本地日志文件（如 `processed_danmaku_files.txt`）记录已经成功处理的弹幕文件 BVID。
   * 每次启动脚本前，先读取记录，自动跳过已处理的文件。
   * 中间计算结果（如词频矩阵、LDA 模型特征）务必使用 `.parquet` 格式或 `pickle` 定期序列化落盘保存，以便崩溃后秒级恢复。
